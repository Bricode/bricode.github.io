Agile end of project Reflection

Our highest priority is to satisfy the customer through early and continuous delivery of valuable software 25/08/2020.

At the end of every sprint, there was a meeting held where we would show the client and class mates our project as it currently stood. As a group, we aimed to have all features finished and functional for this meeting. In this way we aimed to deliver constant valuable software, as the product would always be in a working state with new features. As an individual, I ensured that all user stories asigned to me were fully functional for a presentation to the client. This principle helped keep me accountable and focused becuase the two week looming deadline meant that I had to priorities and not put off doing work after the alotted time. I think this principle was most useful as a driving force after the topic change, as while we lost alot of motivation to work on the software, we still had deadlines to finish features and so we kept working.


Welcome changing requirements, even late in development. Agile processes harness change for the customer’s competitive advantage.

This was the weakest of the agile principles for both me and my group. When this project was started, we were very happy working on features and bringing them up to a high standard. At the start of sprint four when we were told that we would be changing topic, we did not welcome the change as it felt terrible having several features that had lots of effort and time put in be deprecated so suddenly. This had a noticable effect both on team morale and the rate at which we worked on features. If I was to have another run at this project with the hindsight I now have, This would be the principle I would aim to improve on. I dont think I did enough to encourage my team and myself to pick ourselves back up and accept that the change had happened.
